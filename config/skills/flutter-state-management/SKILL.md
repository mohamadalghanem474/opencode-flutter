## What I do

- Identify the active state management pattern in the repo
- Help with controllers, notifiers, blocs, providers, repositories, and services
- Model loading, empty, error, retry, pagination, cache, and cancellation behavior
- Keep transport details out of UI and state ownership explicit

## When to use me

Use this when changing data flow, API integration, local persistence, async orchestration, or user-visible state transitions.

## Operating rules

- Follow the repo's current state pattern first.
- Prefer typed null-safe APIs and predictable async behavior.
- Cover edge states explicitly and update tests when logic changes.
